# Lexos ‚Äî Claude Code Instructions

## What this project is
Lexos is a **vocabulary middleware platform**. It is NOT a voice ordering system.
It sits between a voice workflow and the ASR and LLM engines the workflow already uses.
It does not own the ASR connection, does not stream audio, does not build a custom speech model.
Never build voice ordering features. Only build Lexos infrastructure.

## Current architecture (read before touching intent resolution)
- **Intent resolution = exact match ‚Üí classifier ‚Üí miss.** The semantic stage is a
  trained per-language classifier (`src/classifier/`), NOT nearest-neighbour over
  pgvector. Per-term centroids in `vector_embeddings` are now only an out-of-domain gate.
  See `docs/ARCHITECTURE.md` ¬ß7 and `.claude/rules/decisions.md` for the why.
- **Action-cue gate** (`src/classifier/action-gate.ts`) makes unknown action verbs fail
  safe (‚Üí miss) instead of defaulting to ADD. It uses `action-cues.json` (curated) +
  `vocab.json` (regenerated by the trainer). If you add a new action verb to the corpus,
  it also belongs in `action-cues.json`.
- **Romanized normalizer** (`src/classifier/normalize.ts`) collapses spelling variants
  (dh‚Üíd, dd‚Üíd, aa‚Üía) on both training text and query, so "thegddhu"‚âà"thegdhu". Applied
  before the classifier embedding; native script untouched.
- **Menu grounding (intent + slot)** (`src/menu/`, DB-backed `menu_items`) separates item
  actions from order actions using the tenant's menu. Menu stored one row per **surface form**
  (English + each language + aliases) ‚Üí canonical name. `matchInMenu` (cosine ‚Äî vectors ARE
  right for item synonymy) decides if an item is present. Item present ‚Üí action from cue groups
  (`_item_actions` in `action-cues.json`); no item ‚Üí classifier (order intents) + gate, else miss.
  Returns `matched_item` slot. Ingest: `scripts/ingest-menu.ts <menu.json>` or the admin upload
  (both via `src/menu/expand.ts`). Re-ingest when a menu changes.
- **Admin dashboard** at `/admin` (`LEXOS_ADMIN_KEY`): tenants, menu upload, per-tenant API keys
  (`api_keys`, migration 008 ‚Äî `/enrich` resolves tenant from the key), retrain, gated generation.
  Migrations are applied manually in the Supabase SQL editor.
- **Active languages:** hi-IN, kn-IN, mr-IN, ta-IN, te-IN. bn/gu/es/ar are disabled
  (`active=false`, embeddings purged) ‚Äî toggle via `scripts/toggle-languages.ts`.
- **Eval harness:** `scripts/run-intent-eval.ts <file>` (broad) and `scripts/eval-traps.ts`
  (held-out adversarial). Run after any change to intent resolution.

## Non-negotiables
- Never hardcode config values ‚Äî all constants from environment variables
- All DB queries go through `/src/db/` ‚Äî never raw SQL in route handlers
- Every endpoint has a corresponding test in `/tests/`
- Run tests before marking any task complete
- When unsure about a decision, ask ‚Äî do not guess and proceed
- **NEVER run the Claude/Anthropic API for generation.** Do not run `npm run generate`,
  `npm run supplement`, `npm run supplement:romanized`, or any script that calls the
  Anthropic SDK (`src/generation/`, miss loop, semantic promote loop).
  All vocabulary generation is done by Claude Code (the assistant) directly in the
  conversation ‚Äî not by the Lexos app calling the API at runtime or via scripts.
- **Retrain the classifier after ANY vocabulary change**, then restart the server
  (model is cached in memory): `npx ts-node scripts/train-intent-classifier.ts`.
  This is offline and calls only Cloudflare bge-m3 ‚Äî NOT the Anthropic API, so it is allowed.
  Stale model = wrong intents. Inserting/editing variants alone is not enough.
