Lexos — Deploy & WiseOrder Delivery

Everything needed to go from local to live and send WiseOrder their brief.

1 Deploy to Railway

Why Railway: Auto-detects Node.js, runs your build step, gives you a public HTTPS URL, free tier covers the demo phase, no config files beyond railway.toml (already created).

Step 1 — Push to GitHub

Railway deploys from a Git repo. If you don't have one yet:

# In C:\Users\rayaa\Desktop\lexos
git init
git add .
git commit -m "initial"
# Create a private repo at github.com, then:
git remote add origin https://github.com/<you>/lexos.git
git push -u origin main
⚠ .gitignore check: .env is already excluded. The src/classifier/model.json (trained weights) and src/classifier/vocab.json must be committed — they're not in .gitignore and you need them in production.

Step 2 — Create a Railway project

  • Go to railway.app → New Project → Deploy from GitHub repo
  • Select your lexos repo
  • Railway detects Node.js + your railway.toml. Build command: npm install && npm run build. Start: node dist/index.js

Step 3 — Set environment variables

In Railway: your project → Variables tab. Set all of these:

SUPABASE_URLFrom your .env (already have it)required
SUPABASE_SERVICE_ROLE_KEYFrom your .envrequired

Generate both secrets locally before setting them — keep them somewhere safe, you'll need them for WiseOrder.

Step 4 — Deploy & get your URL

  • Hit Deploy. Railway builds in ~60s
  • Go to Settings → Networking → Generate Domain. You get https://lexos-production-xxxx.up.railway.app
  • Smoke test: curl https://your-domain/health → {"status":"ok","project":"lexos"}

Step 5 — Verify admin dashboard

Open https://your-domain/admin in a browser. You'll need LEXOS_ADMIN_KEY set as the X-Admin-Key header — either via a browser extension like ModHeader, or use the curl approach:

curl https://your-domain/admin/tenants \
  -H "X-Admin-Key: <your-admin-key>"

2 Set up the WiseOrder tenant & API key

Do this once in production after deploy. All via the admin API.

Create the WiseOrder tenant

curl -X POST https://your-domain/admin/tenants \
  -H "X-Admin-Key: <admin-key>" \
  -H "Content-Type: application/json" \
  -d '{
    "tenant_id": "wiseorder",
    "display_name": "WiseOrder",
    "tenant_type": "business",
    "primary_language": "hi-IN",
    "register_preference": "mixed"
  }'

Generate their API key

curl -X POST https://your-domain/admin/tenants/wiseorder/api-keys \
  -H "X-Admin-Key: <admin-key>" \
  -H "Content-Type: application/json" \
  -d '{"label": "wiseorder-production", "product_id": "wiseorder"}'
The key is shown exactly once. Copy the "key" field from the response immediately — this is what you send to the WiseOrder team. It cannot be retrieved again (only the prefix is stored).

Upload a menu (optional but recommended)

If WiseOrder has a specific menu, upload it per-restaurant via the admin API. This unlocks item-level slot resolution (matched_item) and prevents off-menu intent guesses.

curl -X POST https://your-domain/admin/tenants/wiseorder/menu \
  -H "X-Admin-Key: <admin-key>" \
  -H "Content-Type: application/json" \
  -d '{
    "items": [
      {
        "name": "Masala Dosa",
        "category": "BREAKFAST",
        "names": ["मसाला डोसा", "ಮಸಾಲ ದೋಸೆ", "மசாலா தோசை"],
        "aliases": ["masala dose", "masle dose"]
      }
    ]
  }'

3 What to send WiseOrder

Send them

  • The production URL: https://lexos-production-xxxx.up.railway.app
  • Their API key (from step 2)
  • The integration brief: docs/WISEORDER_INTEGRATION.md

They need to do

  • Store the API key as a secret in their environment
  • Call GET /retrieve/asr-bias at session start
  • Call POST /enrich on every utterance
  • Call PATCH /enrich/:id/outcome after each interaction

Update the base URL in the brief before sending

Open docs/WISEORDER_INTEGRATION.md and replace the placeholder at the top:

# Before
https://<lexos-production-url>

# After
https://lexos-production-xxxx.up.railway.app

4 Active languages & coverage

Remind WiseOrder to always pass lang explicitly — the classifier is per-language and wrong/missing lang degrades accuracy.

hi-IN Hindi kn-IN Kannada mr-IN Marathi ta-IN Tamil te-IN Telugu

Disabled (vocabulary retained, can re-enable): bn-IN · gu-IN · es-ES · ar-AE. These return miss until toggled back on via scripts/toggle-languages.ts.

5 Post-deploy checklist

  • Deploy to Railway, verify /health returns 200
  • Set LEXOS_API_KEY and LEXOS_ADMIN_KEY in Railway variables
  • Create the wiseorder tenant via admin API
  • Generate WiseOrder's API key (copy it — shown once only)
  • Fill in the real URL in docs/WISEORDER_INTEGRATION.md
  • Send brief + URL + key to WiseOrder team
  • Upload WiseOrder's menu (optional — improves item slot resolution)
  • Smoke test: POST /enrich with a known Hindi phrase, confirm intent resolves

6 Ongoing ops

If accuracy drops

Add new vocabulary variants as Claude Code in conversation → retrain: npm run train-classifier → restart the Railway deploy (redeploy from dashboard).

If menu changes

Re-upload via POST /admin/tenants/wiseorder/menu. Menu cache invalidates automatically. No restart needed.

If a language needs enabling

Run npm run toggle-languages on bn-IN locally (generates embeddings), commit the updated terms, redeploy.

Admin retrain (remote)

POST /admin/retrain with the admin key — triggers retraining on the server. Status at GET /admin/retrain/status. Hot-reloads model on success.
